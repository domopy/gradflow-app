# 研程 GradFlow：项目上下文

> 本文件用于将前期产品讨论交接给新的 Codex 项目线程。开始开发前，请完整阅读本文件，并查看 `docs/ui-concepts/` 中的三套 UI 概念图。

## 1. 项目概述

### 项目名称

- 中文名：研程
- 英文名：GradFlow
- 仓库名建议：`gradflow`

### 一句话定位

面向研究生的手机日程助手，从聊天截图、课程通知和导师消息中提取考试、作业、组会、实验安排的时间、地点和要求，经用户确认后保存并提醒。

### 核心问题

研究生的重要安排通常散落在微信群聊、导师私聊、课程通知和临时消息里，容易出现：

- 看过消息但忘记记录；
- 忘记考试、作业或材料提交的截止时间；
- 找不到组会或实验安排的地点；
- 消息中的时间、地点后来发生变更；
- “下周”“月底前”“晚上”等模糊信息没有得到确认。

研程不是普通待办清单，也不是泛用会议摘要工具。核心价值是：

> 将非结构化通知变成可核对、可提醒、可追溯的研究生日程。

## 2. 已确认的产品决策

- 产品形态：真正的 Android/iOS 手机 App，不做网页优先方案。
- 开源方式：开源单仓库项目，许可证建议使用 MIT。
- 开发策略：Android 优先测试，同时保持 iOS 兼容。
- 数据策略：本地优先，不要求注册登录，不依赖云数据库。
- 商业模式：当前阶段不考虑。
- AI 策略：用户自行配置兼容 OpenAI 接口的模型服务，采用 BYOK 模式。
- 第一版不自动读取微信聊天，只处理用户主动选择的截图或粘贴的文字。
- AI 结果必须经过人工确认，不能未经确认直接创建日程和提醒。
- 缺失信息不得由 AI 擅自补全，必须标记为“待确认”。

## 3. 目标用户与场景

### 目标用户

- 硕士研究生；
- 博士研究生；
- 同时承担课程、科研、组会和实验任务的学生；
- 经常从聊天消息中接收临时通知的人。

### 首批核心场景

1. 课程群发布考试时间、地点和考试要求；
2. 教师布置作业并说明截止时间和提交方式；
3. 导师临时调整组会时间或地点；
4. 课题组要求组会前准备 PPT、数据或实验结果；
5. 实验室安排设备使用、实验值班或项目任务；
6. 发布讲座、开题、答辩或材料提交通知；
7. 后续消息通知延期、取消、改期或更换地点。

### 示例

输入：

> 明天下午 3 点在实验楼 507 组会，每个人准备 5 页以内 PPT。机器学习作业延期到下周五。

输出候选项：

```text
组会
时间：明天 15:00
地点：实验楼 507
准备事项：5 页以内 PPT

机器学习作业
截止：下周五
变更类型：延期
```

## 4. MVP 功能范围

### 4.1 今日页面

- 展示今天、三天内和本周的重要安排；
- 区分考试、作业、组会、实验、答辩和讲座；
- 展示即将截止、已逾期和待确认事项；
- 可将事项标记为完成；
- 点击卡片进入详情并查看原始证据。

### 4.2 日历页面

- 月历视图；
- 日期上的分类色点；
- 选中日期后展示当天议程；
- 按课程或事项类型筛选；
- 检测时间冲突；
- 将确认后的事项写入手机系统日历。

### 4.3 导入页面

- 从相册选择一张或多张聊天截图；
- 使用相机拍摄通知；
- 粘贴聊天或通知文字；
- 手动添加安排；
- 设置消息发送日期，帮助解析“明天”“下周”等相对时间。

### 4.4 AI 识别与确认页面

- 展示原始截图或文字；
- 展示识别出的候选事项；
- 每条事项可单独保存、编辑或忽略；
- 显示置信度；
- 显示对应原文证据；
- 突出显示待确认字段；
- 支持一次保存多条安排。

### 4.5 本地提醒

- 为考试、作业截止和其他日程设置本地通知；
- 支持默认提前一天、提前一小时等规则；
- 编辑或删除事项时同步更新通知；
- 第一版不开发云端推送服务。

### 4.6 设置页面

- AI 服务地址；
- API Key；
- 模型名称；
- 默认时区；
- 默认提醒规则；
- 原图保留策略；
- JSON 数据导入与导出；
- 隐私说明。

## 6. AI 提取规则

AI 应提取：

- 标题；
- 事项类型；
- 所属课程或项目；
- 开始时间；
- 截止时间；
- 地点；
- 提交方式；
- 准备材料或要求；
- 相关人员；
- 原始时间表达；
- 原文证据；
- 置信度；
- 待确认字段；
- 是否属于现有安排的变更。

必须遵循以下规则：

1. 不得虚构消息中不存在的信息；
2. “尽量”“应该”“回头”“找时间”等表达不能视为确定安排；
3. 没有具体时间时保留原始表达，并标记待确认；
4. 相对日期必须结合消息发送日期、用户时区和当前学期解析；
5. 一条消息可提取多个事项；
6. 必须识别延期、取消、改期和地点变更；
7. 每项结论必须能回溯到原始文本；
8. AI 输出必须通过运行时 Schema 校验；
9. AI 结果必须经用户确认后才能保存。

### 建议的结构化输出

```json
{
  "items": [
    {
      "type": "meeting",
      "title": "课题组组会",
      "course": null,
      "startAt": "2026-06-21T15:00:00+08:00",
      "dueAt": null,
      "location": "实验楼507",
      "submissionMethod": null,
      "requirements": ["准备5页以内PPT"],
      "sourceQuote": "明天下午3点在实验楼507组会",
      "originalTimeText": "明天下午3点",
      "confidence": 0.96,
      "uncertainFields": [],
      "changeType": "created"
    }
  ]
}
```

## 7. 技术方案

### 推荐技术栈

```text
Expo React Native
TypeScript
Expo Router
Expo SQLite
Expo Notifications
Expo Image Picker
Expo Calendar
Expo SecureStore
Zod
Vitest
```

### 技术原则

- 使用单仓库；
- 不单独开发后端；
- 不引入 Redux，优先使用 React Context、Hooks 和小型状态模块；
- SQLite 保存结构化数据；
- SecureStore 只保存 API Key 等敏感配置；
- 不把 API Key 写入 SQLite、日志、代码仓库或导出文件；
- 服务层与 UI 分离，AI 模型提供商通过适配器接入；
- 不引入新的生产依赖，除非确有必要并经过确认；
- 遵循项目 `AGENTS.md` 中的编码、注释、测试和安全规范。

### AI 处理流程

第一版优先使用支持图像理解的模型：

```text
用户选择截图
→ App 压缩图片
→ 发送给用户配置的模型服务
→ 返回结构化 JSON
→ Zod 校验
→ 用户编辑和确认
→ 写入 SQLite
→ 创建本地通知
```

当前图片识别流程：

```text
用户确认后上传压缩图片到 EasyOCR
→ EasyOCR 返回识别文字
→ 只向 DeepSeek 发送文字
→ 返回结构化结果
```

## 10. UI 方向

三套概念图位于：

- `docs/ui-concepts/concept-a-academic.png`
- `docs/ui-concepts/concept-b-dark.png`
- `docs/ui-concepts/concept-c-notebook.png`

### A：清爽学术

- 白色和暖灰背景；
- 深海军蓝为主色；
- 信息层级清楚；
- 最适合正式主方案；
- 开发难度适中，适合长期使用。

### B：深色星轨

- 深色背景和轨道式时间展示；
- 视觉冲击最强；
- 置信度和结构化解析展示非常优秀；
- 不建议直接作为唯一主题，长期使用可能显得偏重。

### C：温暖手帐

- 奶油色、陶土色和纸张质感；
- 更有人情味；
- 卡片式任务呈现自然；
- 需要克制手帐元素，避免幼稚或过度拟物。

### 推荐组合

采用：

- A 的信息架构与整体布局；
- C 的温暖色彩和轻微卡片质感；
- B 的置信度展示方式；
- 避免渐变堆叠、重玻璃效果和过多装饰；
- 保证中文字号、对比度和触控区域符合移动端可用性要求。

概念图仅用于视觉方向，不应逐像素照抄。正式实现需要建立可复用的设计令牌：

```text
颜色
字号
字重
间距
圆角
阴影
图标尺寸
触控区域
状态颜色
```

## 11. 隐私与安全

- 只读取用户主动选择的截图；
- 不请求整个相册的长期访问权限；
- 上传截图前明确提示会发送给用户配置的模型服务；
- 默认允许用户在解析后删除原图，仅保留结构化事项和证据文字；
- API Key 存入 SecureStore；
- 日志中不得打印 API Key、完整聊天内容或图片 Base64；
- 数据导出默认不包含 API Key；
- 仓库不得提交 `.env`、SQLite 数据库、真实截图或真实聊天记录；
- 示例和测试必须使用虚构数据；
- 删除事项时同步处理相关提醒；
- 删除全部数据时应清理本地图片、数据库记录和通知。

## 12. 测试重点

日期和变更识别是项目最大的风险，应重点覆盖：

- 今天、明天、后天；
- 本周五、下周三、下下周；
- “6月30日晚12点”；
- “月底前”“开学第一周”；
- “晚上”“稍后”“找时间”等模糊表达；
- 一条消息包含多个安排；
- 跨年份日期；
- 不同用户时区；
- 原定周三考试，后来改为周五；
- 作业延期；
- 组会取消；
- 地点不变或只修改地点；
- 编辑事项后重新调度通知；
- AI 返回非法 JSON 或缺失字段；
- 权限被拒绝、模型请求失败和离线状态。

## 14. MVP 成功标准

核心验证目标：

> 用户上传一张群聊截图，在 30 秒内得到准确、可编辑、可提醒的安排。

建议指标：

- 首次导入流程可在 60 秒内完成；
- 候选事项接受率达到 80% 以上；
- 重要字段错误时用户可以快速修正；
- 所有 AI 结论都能查看原文；
- 没有明确时间的信息不会被伪造成精确日程；
- 编辑、删除后不存在失效或重复通知。

## 15. 暂不实现

- 自动读取微信全部聊天；
- 微信机器人；
- 用户账户和云同步；
- 团队协作与复杂权限；
- CRM 或完整项目管理；
- 自动替用户发送消息；
- AI 会议录音机器人；
- 商业订阅和支付；
- 十几个模型或平台的复杂集成。
