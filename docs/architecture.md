# v1.0架构说明

## 技术边界

研程使用Expo React Native、TypeScript、Expo Router和Expo SQLite构建。应用没有自建后端，当前发布与验证范围为Android，Web用于开发预览。

## 目录职责

- `src/app/`：路由和页面；
- `src/components/`：可复用界面组件；
- `src/db/`：数据库迁移和Repository；
- `src/providers/`：页面共享的日程状态与业务操作；
- `src/services/`：系统通知和系统日历适配；
- `src/schemas/`：运行时数据校验；
- `src/theme/`：颜色、间距、圆角、字号和类型视觉令牌；
- `src/types/`：领域类型；
- `src/utils/`：无副作用的日期与分组逻辑。

页面不直接执行SQL，也不直接管理通知记录。`ScheduleProvider`负责协调Repository、通知服务和界面刷新。

## AI文字提取

`src/services/ai/`包含DeepSeek配置、提示词和Chat Completions适配器：

1. 用户主动粘贴文字并设置消息发送时间；
2. 客户端从SecureStore读取DeepSeek API配置；
3. 请求使用JSON Output模式，并显式要求只返回JSON；
4. 返回值先解析JSON，再通过Zod运行时Schema校验；
5. 候选事项进入内存确认会话，不直接写入数据库；
6. 用户选择保存后，在同一个SQLite事务中创建source和所有日程。

Web开发预览使用内存数据与模拟提取结果，不发送真实AI请求。移动端使用真实SQLite和DeepSeek接口。

## 图片提取

图片导入复用文字提取与确认管线，在其前增加EasyOCR云识别：

1. 用户从系统相册选择最多5张图片，或主动拍摄通知；
2. 图片最长边压缩到1600像素，生成临时JPEG；
3. 用户确认后，客户端用`multipart/form-data`逐张上传EasyOCR；
4. 请求通过`X-Access-Key`鉴权，返回的`words[].text`按图片顺序合并；
5. 合并后的OCR文字发送到DeepSeek兼容接口；
6. 候选事项仍需通过Zod校验并由用户逐项确认；
7. 默认删除临时图片；若用户选择保留，则将压缩副本复制到应用私有目录并记录到source。

EasyOCR配置与DeepSeek配置相互独立，两种密钥分别存入SecureStore。客户端统一处理无效密钥、额度不足、上传校验、服务不可用和超时。图片OCR不再依赖自定义原生模块，因此可在Expo Go中调试。

## 数据一致性

- 删除事项前先取消对应系统通知，数据库外键随后级联删除提醒记录；
- 编辑事项时，只有用户明确修改提醒选项才重新调度通知；
- 识别确认页可为新安排、改期、延期和换地点直接选择提醒提前量；变更事项默认保持原提醒，取消事项自动关闭提醒；
- 提醒优先以事项开始时间为目标，没有开始时间时才使用截止时间；
- Android精确定时提醒声明`SCHEDULE_EXACT_ALARM`，原生权限变更后需要重新构建安装包；
- SQLite启用外键和WAL模式；
- 数组字段以JSON字符串保存，并在读取时通过Schema校验。
- 删除某个source关联的最后一项日程时，同时删除该source保留的压缩图片。

## 日期与时间输入

`DateTimeInput`统一承载导入、识别确认、新建和编辑页面的日期时间输入。移动端使用`@react-native-community/datetimepicker`提供中文日期与24小时制时间选择器，同时保留`YYYY-MM-DD HH:mm`文本输入；Web预览只保留文本输入。

日期和时间分开合并：选择日期时保留当前文本中的时分，选择时间时保留年月日。选择器每次打开前重新解析文本，只有用户确认后才写回，因此取消操作或尚未完成的手动输入不会被覆盖。最终保存时仍由统一解析函数校验无效日期，并转换为ISO时间写入领域模型。

不完整的手动文本不会作为选择器回退值使用。组件会要求用户先完成或清空输入，避免系统选择器用当前时间覆盖尚未编辑完的内容。

## 系统日历同步

数据库v4在事项上保存当前设备的系统日历事件ID。首次写入创建事件；再次同步或编辑事项时更新同一事件；删除事项时尝试删除系统事件。若用户已在系统日历中删除事件，下次同步会重新创建。事件ID具有设备局部性，不进入JSON备份。

## 备份文件

备份内容仍由`backup-service`负责Schema校验和事务恢复，`backup-file-service`只负责`.json`文件写入、系统分享和文件选择。两层分离使数据格式测试不依赖平台文件API。

## 事项变更

提取请求会向模型提供最多100条未取消事项的标识、标题、课程、时间和地点。模型对变更消息返回`relatedItemId`候选，用户在确认页最终决定关联关系。

确认后不会创建重复事项，而是按变更类型更新原事项：

- `rescheduled`更新明确提供的开始或结束时间；
- `extended`更新截止时间；
- `relocated`更新地点；
- `cancelled`保留原内容并更新为取消状态。

每次变更在`schedule_changes`保存前后快照和本次source。原事项已有提醒时，改期和延期沿用原提前量重新调度；取消则删除提醒。

## 冲突检测

冲突检测只处理具有开始时间且未完成、未取消的事项。没有明确结束时间时按一小时估算；作业截止时间不被视为持续占用区间，避免从开始到截止产生跨天误报。

## JSON备份

备份格式具有固定标识`gradflow-backup`和版本号`1`，包含sources、schedule_items和schedule_changes。API Key、系统通知标识和图片文件不进入备份。恢复前先完整校验，再以事务替换本地数据；校验失败不会修改数据库。

## 后续扩展

后续版本可增加文件选择器、增量合并恢复和更灵活的默认提醒规则。
